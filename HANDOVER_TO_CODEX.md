## 3. 数据模型（最重要，看懂这节就能维护）

所有数据内嵌在 index.html 和 data.js 里，是 JS 字面量。四个核心结构：

### 3.1 `FX` — 赛程（数组）
```js
const FX=[
 ["2026-06-13T03:00:00+08:00","A","mx","za","墨西哥城"],  // [北京时间ISO, 组/轮次, 队1code, 队2code, 球场中文]
 ...
 ["2026-07-04T06:00:00+08:00","R32","ar","cv","迈阿密"],   // 淘汰赛已解析
 ["2026-07-05T01:00:00+08:00","R16","w73","w75","休斯顿"], // 淘汰赛未解析(签位)
];
```
- 第 2 位 = **组名(A-L)** 或 **淘汰赛轮次码**：`R32 / R16 / QF / SF / P3(季军) / FIN(决赛)`。
  - ⚠ **决赛码是 `FIN` 不是 `F`**（`F` 会和 F 组撞码，是踩过的坑，切勿改回 `F`）。
- 队 code：ISO 风格小写（`es`西 `ar`阿 `eng`英格兰 `gb-sct`苏格兰 `cd`刚果金 `cv`佛得角…）。名字/旗帜从 `ZH/EN/FLAG/EMOJI` 映射查。
- **淘汰赛签位码**（team1/team2 位置）：`1X`=X组头名、`2X`=X组次名、`3XYZ`=X/Y/Z等组的第三名、`wN`=第N号比赛胜者、`lN`=第N号比赛负者。渲染时 `koLabel()` 把它们显示成"A组②/M73胜"等；解析成真实队 code 后自动显示球队+旗帜+预测。

### 3.2 `KNOWN` — 真实比分（对象，最高优先级）
```js
const KNOWN={
 "mx|za":[2,0], ...
 "za|ca":[0,1], "de|py":[1,1],  // 淘汰赛。key 朝向 = FX 里的 队1|队2
};
```
- key = `"队1|队2"`（**必须与 FX 里该场的队顺序一致**）。value = `[队1进球, 队2进球]`。
- **点球淘汰**：记 90/120 分钟比分（如 1-1 记 `[1,1]`），晋级方在注释 + PREDICTIONS.md 里标注（如 `// nl|ma 1-1摩洛哥点球晋级`）。站点只显示这个比分。

### 3.3 `PRED` — 三轨预测（对象）
```js
"ar|cv":{
  pred:[2,0],                    // 九维(十一维基本面)预测比分 [队1,队2]，90分钟
  alt:"次选 1-0 / 3-0",          // 备选
  conf:"中",                     // 信心: 低/中低/中/高
  prob:[59,31,11],               // 胜平负% [队1胜,平,队2胜]，三者和必须=100
  dims:[[code,note], ... 共11个], // 十一维独立分析，见 §3.4 和 §6
  odds:{winner:"ar",prob:[59,31,11],conf:"中",move:"...",sharp:"..."}, // 市场轨
  x:{ht8:[1,0],os:[3,0],oht:"ar",oou:"over"},  // 市场衍生: 半场比分/市场比分线/半场赢家/大小球
  upset:{pred:[0,1],winner:"cv",ou:"under",prob:18,why:"..."}, // 爆冷轨(winner=""表示预测平局)
  verdict:"..."                  // 综合裁决(人读)
}
```
- **朝向铁律**：`pred` / `prob` / `x.os` / `x.ht8` 都按 `[队1,队2]` 朝向，队顺序 = FX/key 里的顺序。
- `upset.winner`：弱旅 code（弱旅晋级）或 `""`（预测平局，淘汰赛即拖点球）。
- 没有 odds/x/upset 的场次会优雅降级（渲染跳过对应轨）。

### 3.4 `DIMS` — 维度标签（11 个，顺序固定）
```js
const DIMS=["首发阵容","伤停情况","历史底蕴","历史交锋·同大洲","最近状态",
            "裁判尺度","气温环境","教练战术","战略卡位","控球率","攻防数据"];
```
- PRED 的 `dims` 数组按此顺序，每项 `[code, note]`：`code`=该维度倾向的队(或`""`中性)，`note`=解读文字。
- ⚠ **历史场次(小组赛)只有 9 维**，渲染层自动补 2 条中性占位，不用管。新预测必须写满 **11 维**。

### 3.5 命名映射
`ZH`(中文名) / `EN`(英文名) / `FLAG`(旗帜) / `EMOJI`(emoji旗) / `N2C`(名→code) 都在 data.js / index.html 里。新队 code 必须在这些映射里存在，否则显示异常。淘汰赛全是已参赛队，无需新增。

## 4. 三轨预测体系

每场三条**独立**预测线，赛后分轨评分：

| 轨 | 字段 | 含义 | 评分口径 |
|---|---|---|---|
| **① 十一维基本面** | `pred`/`prob`/`dims` | 十一维分析 → λ 机制算比分 | 赢家方向命中 / 比分全中 |
| **② 市场** | `odds`/`x` | 赔率 favorite + 市场比分线 | favorite 赢家命中 / 市场比分线全中 |
| **③ 爆冷** | `upset` | 尾部冷门(弱旅晋级/点球) | **只在"热门翻车"场计分**：热门没赢时，爆冷是否说中方向/比分 |

> 站点顶部有三轨×(赢家+分数)六指标实时统计条。**爆冷轨用"翻车场"做分母**（尾部 call 的本职是抓另两轨会漏的冷门），这是刻意设计，别改成全场次分母。
> 历史品牌名叫"九维"，现已升级"十一维"，UI 已改名。verdict 里个别旧场次仍写"九维"属历史正确。

## 5. λ 比分机制（可复现，必须照此生成九维比分）

**核心**：不再拍脑袋套比分，而是用泊松 + Dixon-Coles 从"预期进球 λ"算出最可能比分。这样比分按对位分布（强弱悬殊→3-1，双铁桶→0-0），而非千篇一律 2-1。

**流程**：
1. 各队用**此前真实攻防率**：单场进球**封顶 4**（防 7-1 污染）+ 向联盟均值 μ **收缩**（k≈1.6，降小样本噪声）。
2. `att_i = (GF_i + μ·k)/(场次 + k)`，`def_i = (GA_i + μ·k)/(场次 + k)`；`α_i = att_i/μ`，`β_i = def_i/μ`。
3. `λ1 = α1·β2·μ`，`λ2 = α2·β1·μ`；**淘汰赛 ×0.93**（更谨慎）；**真东道主主场 +0.12~0.15**（墨/美在本国球场）。
4. 泊松网格 `P(i,j)=pois(i,λ1)·pois(j,λ2)`，对 (0,0)(1,0)(0,1)(1,1) 施 **Dixon-Coles 修正**（ρ≈-0.13，抬高低分平局概率）。
5. **取"最可能结果(胜/平/负)内的最可能比分"** 作 `pred`（不能直接取全局 argmax——那样最可能精确比分总是 1-1，是泊松陷阱）；胜平负% 作 `prob`（凑整=100）。
6. **小样本失真允许判断覆盖**：如阿根廷 λ 被佛得角两个 0-0 压低到 1.36（模型给 1-0），但卫冕冠军质量非 3 场能体现 → 人工上调到 2-0，并在 verdict 注明"判断覆盖模型"。

**可直接运行的参考实现**（把某轮球队和已知比分喂进去即可）：
```js
// node 片段。gp[team]={gf,ga,p} 为各队此前累计(封顶4后)进/失/场次
const fact=n=>{let r=1;for(let i=2;i<=n;i++)r*=i;return r;};
const pois=(k,l)=>Math.exp(-l)*Math.pow(l,k)/fact(k);
const RHO=-0.13;
function model(l1,l2){const M=7;let W=0,D=0,L=0,bw=[0,0],bwp=0,bd=[0,0],bdp=0,bl=[0,0],blp=0;
  for(let i=0;i<M;i++)for(let j=0;j<M;j++){let p=pois(i,l1)*pois(j,l2);
    if(i===0&&j===0)p*=1-l1*l2*RHO;else if(i===1&&j===0)p*=1+l2*RHO;
    else if(i===0&&j===1)p*=1+l1*RHO;else if(i===1&&j===1)p*=1-RHO;
    if(i>j){W+=p;if(p>bwp){bwp=p;bw=[i,j];}}else if(i===j){D+=p;if(p>bdp){bdp=p;bd=[i,j];}}
    else{L+=p;if(p>blp){blp=p;bl=[i,j];}}}
  const s=W+D+L;W=W/s*100;D=D/s*100;L=L/s*100;
  const top=Math.max(W,D,L),sc=top===W?bw:top===D?bd:bl;
  return{score:sc,prob:[Math.round(W),Math.round(D),Math.round(L)]};}
// λ 估计：μ=联盟场均(每队), 见 §5 步骤2-3
```
> `market` 轨(odds)与 `upset` 轨(爆冷)沿用既有框架人工写，不走 λ。

## 6. 十一维 dims "三段式深写"标准（硬要求）

每个维度的 `note` **≥50 字**，含三段：
1. **具体**：球员(带俱乐部/身价)、真实数字、阵型
2. **机制**：这一项为什么影响本场
3. **所以呢**：胜负手 / 盲区 / 反向风险

各维度数据来源与写法：
- **控球率**：用**真实联网值**（FBref / FIFA / ESPN 逐场平均）。注明"控球≠强弱"（高控可能无效控球、防反低控照样赢）。
- **攻防数据**：小组赛真实场均进/失（净胜球）。这是 λ 的输入，也是最强信号。
- **历史交锋·同大洲**：有近年直接交锋就用直接战绩；**无交锋则用"同大洲代理"**（如巴西-日本无交锋→查巴西 vs 韩国/亚洲队），注明是代理。
- **伤停情况**：**没有实时伤病源就写"需赛前核实 + 谁缺阵会怎样"，绝不硬编具体伤情**（守 grounding 红线）。
- 球员名只写有把握在 2026 阵中的球星；冷门队(佛得角/刚果金)多用打法+真实数据，不杜撰名单。

## 9. 已知坑 / 边界（务必知道）

1. **两文件同步**：KNOWN/PRED/FX/DIMS 改动必须同时改 index.html 和 data.js，同值。只改一个会导致小程序数据与网页不一致。
2. **决赛码 `FIN` 不是 `F`**（撞 F 组）。类似地新增任何轮次码别用单个 A-L 字母。
3. **scores.json 已 stale**（停在 6/28）→ 淘汰赛结果靠 WebSearch，别信 scores.json。
4. **点球场评分**：KNOWN 只存 90 分钟比分（平局），晋级方靠注释/PREDICTIONS.md 记。自动按"90分钟胜负"评爆冷会漏记点球晋级的命中，需手动认定。
5. **λ 小样本失真**：3 场小组数据会失真（强队被弱队铁桶压低、或屠鱼腩虚高），允许判断覆盖 + verdict 注明。λ 也吃不到"超级个体(哈兰德级)淘汰赛一击定胜负"的权重——这类场次别盲信 λ。
6. **prob 必须和=100**（渲染概率条 + 评分依赖）。
7. **签位解析要对**：R32 的第三名分配遵循 FIFA 预定表；解析下一轮时，用"本轮胜者"填 wN、"负者"填 lN，别搞反朝向。
8. **爆冷别都写 1-1/平局**（历史教训：用户会质疑"爆冷都是小球平局？"）。要按对位给多样比分：弱旅赢(2-1)、铁桶点球(0-0)、对攻(2-2)等，混合。
9. **对客/公开性质**：这是给个人+小圈子看的娱乐站，非严肃博彩。措辞留娱乐/免责口径。
10. **原自动化**：曾用 Claude Code 的 durable cron（每天 8:31）跑此例程。移交给你后，请用你侧的定时机制（cron/launchd）承接，别依赖那个（recurring 7 天过期）。
